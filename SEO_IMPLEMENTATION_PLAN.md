# SEO Implementation Plan — diego-aguirre.com

**Prepared:** April 2026  
**Branch:** `feature/about-page-carousel-and-builds-teaser`  
**Audited by:** Senior SEO Strategist + Staff Product Engineer  
**Status:** Audit complete — no changes implemented yet

## Executive Summary

The site has a strong technical SEO foundation. Next.js 15 App Router with static export, canonical tags on every page, a dynamic sitemap, JSON-LD schemas, and correct OG/Twitter cards are all in place and working. This is meaningfully ahead of the average personal site.

The gaps are surgical, not structural. The highest-leverage fixes are:

1. **BreadcrumbList schema** on build detail pages — free rich result, zero visible change
2. **GitHub profile missing from `Person.sameAs`** — direct entity disambiguation gap
3. **`SoftwareApplication` schema using non-standard values** — blocks rich result eligibility
4. **Sitemap `lastModified` always equals build time** — degrades freshness signal quality
5. **`llms.txt`** — low effort, rapidly growing AI-search surface
6. **Missing `Offer` on SoftwareApplication** — required for Google to unlock rich snippets
7. **Internal trailing-slash redirects** on several in-page links — crawl budget micro-tax

None of these touch visible content, layout, or design. All are static-export safe.

**Estimated total implementation time: 3–5 focused hours.**

## Problems Found

### P1 — Critical Impact

#### P1.1 — No BreadcrumbList Schema on Nested Pages

**File:** `lib/structured-data.ts`  
**Affected:** `/builds/narrative/`, `/builds/craiive/`, `/builds/nestiq/`

BreadcrumbList is the easiest structured data rich result to unlock on a site with hierarchical URLs (`/builds/[slug]/`). Google explicitly shows breadcrumbs in the SERP snippet when BreadcrumbList is present. It costs zero visible change and takes ~15 minutes to implement.

Currently there is no BreadcrumbList schema anywhere on the site. Every build detail page should declare:  
`Home > Builds > {Project Name}`

#### P1.2 — GitHub Profile Absent from `Person.sameAs`

**File:** `lib/site.ts`, line 78–83  
**Schema:** `createPersonSchema()` in `lib/structured-data.ts`, line 23

The `githubProfileLink` variable in `lib/site.ts` is **named** github but contains a LinkedIn URL:

```ts
// lib/site.ts:78-83
const githubProfileLink: SiteLink = {
  href: "https://www.linkedin.com/in/diego-aguirre-110729219/",  // ← LinkedIn, not GitHub
  label: "Contact",
  ...
};
```

As a result, the `Person` schema `sameAs` array contains LinkedIn and Instagram but **no GitHub profile URL**. For a software engineer, the GitHub profile is the highest-authority identity signal for entity disambiguation in Google's Knowledge Graph. This directly affects whether Google can confidently associate "Diego Aguirre" with a software engineering entity.

The GitHub profile URL (`https://github.com/insdaguirre`) is displayed on the builds page (in link href), but it is absent from the structured data where it matters most.

#### P1.3 — `SoftwareApplication.applicationCategory` Uses Non-Standard Values

**File:** `lib/structured-data.ts`, line 93  
**Data source:** `builds-content.ts`, `productType` field per project

```ts
// Current values:
"Higher education intelligence platform"  // Narrative
// (likely similar non-standard strings for Craiive, NestIQ)
```

Schema.org's `SoftwareApplication.applicationCategory` has a list of expected values including `"BrowserApplication"`, `"WebApplication"`, `"MobileApplication"`, `"DesktopApplication"`, etc. Non-standard freeform strings are not recognized by Google's Rich Results parser and may silently prevent rich result eligibility.

**Fix:** Use `"WebApplication"` (primary) + add `"applicationSubCategory"` for the domain description.

#### P1.4 — `creativeWorkStatus` Uses Non-Standard Values

**File:** `lib/structured-data.ts`, line 100  
**Data source:** `builds-content.ts`, `status` field

```ts
...(status ? { creativeWorkStatus: status } : {}),
// status = "LIVE" — not a schema.org CreativeWorkStatus value
```

Schema.org's `CreativeWorkStatus` expects values from its vocabulary: `"Draft"`, `"Published"`, `"Obsolete"`, `"Discontinued"`. The value `"LIVE"` is not a valid schema.org term. Validators will flag this. Fix: map "LIVE" → `"Published"`, or remove the field.

### P2 — High Impact

#### P2.1 — `sitemap.lastModified` Is Always Build Time

**File:** `app/sitemap.ts`, line 31

```ts
const lastModified = new Date();  // Always "right now" at build time
```

Google uses `lastModified` to prioritize re-crawling. When every page always reports the current timestamp, the signal is noise — Googlebot can't distinguish which pages actually changed since the last crawl. This matters more as the site grows.

**Fix:** Add a `lastModified` date to each route definition (or each `BuildProject` in `builds-content.ts`) and use those static dates.

#### P2.2 — `CollectionPage` ItemList Uses External URLs

**File:** `lib/structured-data.ts`, lines 140–149  
**Schema:** `createCollectionPageSchema()` used on `/builds/`

The `ItemList` in the builds page schema maps items to external URLs (GitHub repos, live product URLs) rather than the internal canonical `/builds/[slug]/` pages. This dissipates internal link equity in structured data and misrepresents the collection's item URLs to Google.

**Fix:** Map `item.url` to the internal `/builds/[slug]/` path for current build projects.

#### P2.3 — No `Offer` on `SoftwareApplication` Schema

**File:** `lib/structured-data.ts`, lines 89–103

Google's structured data guidelines for `SoftwareApplication` strongly recommend an `offers` property to indicate pricing. Without it, rich results for software may not trigger. For free-tier apps, a zero-price `Offer` is the correct declaration.

```json
// What should be added:
"offers": {
  "@type": "Offer",
  "price": "0",
  "priceCurrency": "USD"
}
```

#### P2.4 — `Person.image` Points to OG Preview, Not a Portrait

**File:** `lib/structured-data.ts`, line 20

```ts
image: absoluteUrl(siteConfig.socialImage),  // /og/diego-aguirre-preview.png (3024x1724)
```

The `Person` schema `image` should ideally be a portrait/headshot photo. Google uses this when generating Knowledge Graph cards and entity cards. A full website screenshot (`3024x1724`) is not a person photo. This doesn't block indexing but weakens entity richness.

**Fix (conditional):** If a headshot image exists or is added to `public/`, update the schema to reference it. This is a soft recommendation.

### P3 — Medium Impact

#### P3.1 — `og:type` Is Always `"website"` for Project Pages

**File:** `lib/site.ts`, line 231 (`createPageMetadata()`)

The Open Graph protocol has specific types for content: `article`, `website`, `profile`, `book`, etc. Build detail pages are not generic websites — they represent specific works. While Google doesn't weight this heavily for ranking, Facebook/LinkedIn/Slack unfurls benefit from correct `og:type`.

**Note:** Changing this requires modifying `createPageMetadata()` to accept an optional `ogType` parameter. Low risk but touches a shared utility.

#### P3.2 — Internal Links Missing Trailing Slashes

**Files:** `components/homepage/IdentitySection.tsx` (lines ~85, 89), `components/about/AboutHero.tsx` (line ~46)

Links to `/builds` and `/about` (without trailing slash) cause 301 redirects since `trailingSlash: true` is set. These are in in-page CTAs, not navigation. The navigation links in `lib/site.ts` are correct. Each redirect wastes a small amount of crawl budget and dilutes link signals marginally.

#### P3.3 — No `llms.txt` File

**Files to create:** `public/llms.txt`

`llms.txt` is an emerging web standard (adopted by ~10% of prominent sites as of 2025) that guides LLM crawlers (ChatGPT, Perplexity, Claude, Gemini) on how to parse and cite a site. AI-search referrals grew 3.26x YoY in 2025 and structured data receives 3–5x higher retrieval priority in AI-generated responses. For a personal brand/portfolio, being citable in AI search is increasingly high-value.

`llms.txt` is a plain text or markdown file at `/llms.txt` — it's free, zero-risk, static-export compatible, and takes 20 minutes to write.

#### P3.4 — robots.txt Has Non-Standard `Host:` Directive

**File:** `app/robots.ts`, line 13

The `host: siteConfig.url` property in Next.js robots response generates:
```
Host: https://diego-aguirre.com
```

This is a Yandex-specific directive. Google does not recognize it. It's harmless but technically incorrect for a standards-compliant `robots.txt`. The fix is to remove `host` from the robots response.

#### P3.5 — Missing `datePublished` / `dateModified` on Project Schemas

**File:** `lib/structured-data.ts`, lines 78–104  
**Data source:** `builds-content.ts` — no date fields on `BuildProject`

Project pages have no temporal signals in their schemas. Adding `datePublished` and `dateModified` to the `WebPage` schema gives Google freshness context, which is increasingly valued by AI-search systems.

### P4 — Low Impact / Polish

#### P4.1 — No `potentialAction` (SearchAction) on `WebSite` Schema

**File:** `lib/structured-data.ts`, `createWebSiteSchema()`

A `SearchAction` on the `WebSite` schema enables Google Sitelinks Search Box. The site doesn't have internal search, so this is moot for now. However, if search is ever added, this is the schema hook.

#### P4.2 — Only SVG Favicon

**File:** `public/icon.svg`, referenced in `app/layout.tsx` line 40–44

An SVG-only favicon works in modern browsers but some social crawlers, bookmark tools, and older systems need `favicon.ico`. Adding `public/favicon.ico` (32x32) and a `public/apple-touch-icon.png` (180x180) improves mobile bookmark appearance and PWA readiness.

#### P4.3 — `Person.knowsAbout` Could Be More Specific

**File:** `lib/site.ts`, lines 157–163

Current values: `["AI products", "Product engineering", "Workflow systems", "Decision surfaces", "Founder-led execution"]`

These are brand-voice phrases, not searchable technical keywords. Google's Knowledge Graph benefits from recognizable entity terms. Consider adding: `"Next.js"`, `"TypeScript"`, `"LLM integration"`, `"SaaS"`. This is non-visible and safe.

#### Fix 1: Add BreadcrumbList JSON-LD to all build detail pages

**SEO ROI Score: 5.0**  
`ROI = (Impact:5 × Probability:5) / (Time:1 × Complexity:1 × Risk:1) = 25`

**Impact:** BreadcrumbList unlocks rich breadcrumb display in Google SERPs (the `Home > Builds > Project` path under the title). This improves click-through rate and signals content hierarchy to Google. It's one of the most reliable rich results for nested-URL sites.  
**Probability:** Extremely high — site structure is perfect for it (`/builds/[slug]/`).

**Files to change:**
- [`lib/structured-data.ts`](lib/structured-data.ts) — add `createBreadcrumbSchema()` factory
- [`app/builds/[slug]/page.tsx`](app/builds/%5Bslug%5D/page.tsx) — inject BreadcrumbList alongside existing schemas

**Exact change — `lib/structured-data.ts` (add after line 104):**
```ts
interface BreadcrumbItem {
  name: string;
  url: string;
}

export function createBreadcrumbSchema(items: BreadcrumbItem[]) {
  return {
    "@context": "https://schema.org",
    "@type": "BreadcrumbList",
    itemListElement: items.map((item, index) => ({
      "@type": "ListItem",
      position: index + 1,
      name: item.name,
      item: item.url,
    })),
  };
}
```

**Exact change — `app/builds/[slug]/page.tsx`:**
```tsx
// In the JSX, update the JsonLd call to include breadcrumbs:
<JsonLd
  data={[
    createProjectPageSchema({ ... }),
    createBreadcrumbSchema([
      { name: "Home", url: absoluteUrl("/") },
      { name: "Builds", url: absoluteUrl("/builds/") },
      { name: project.name, url: absoluteUrl(path) },
    ]),
  ]}
/>
```

**Static-export safe:** Yes — pure JSON-LD, no server runtime.  
**Affects visible UI:** No.  
**Test plan:** Build locally → open `/out/builds/narrative/index.html` → confirm `BreadcrumbList` script tag present → run Google Rich Results Test on the deployed URL.  
**Rollback:** Revert the two file changes.

#### Fix 14: Improve image format delivery (WebP conversion)

**Context:** `images: { unoptimized: true }` disables Next.js image optimization for static export. All screenshots are served as PNG (potentially 1–3 MB each).

**Options for static export:**
1. **Cloudflare Pages (free)** — Deploy via Cloudflare instead of GitHub Pages. Cloudflare's CDN serves images through Polish (automatic WebP conversion, free on all plans). Zero code change required — just a hosting change.
2. **Pre-convert at build time** — Add a `scripts/optimize-images.mjs` build step using `sharp` to convert `guis/*.png` → `guis/*.webp` and update imports. Higher complexity.
3. **Cloudflare Images** (paid) — Out of scope for near-zero-cost constraint.

**Recommendation:** Option 1 (Cloudflare Pages) gives the most SEO gain for zero engineering effort — faster LCP, better Core Web Vitals, same zero cost. GitHub Pages won't be missed.

**Risk:** DNS/deployment change. Requires updating CNAME and deployment target. Low technical risk, moderate operational risk (test before cutting over).
